MAIN RESEARCH DESIGN
Build a revised path-planning and rerouting framework for Dhaka with the following ideas:

A. STEADY-STATE / NORMAL NETWORK PREPARATION
1. Build a road graph with road class, width, and condition information.
2. Add vehicle-category access constraints on edges.
3. For every source-destination pair and every relevant vehicle category, compute stable feasible routes.
4. Build a combined stable network, not only isolated pair solutions.
5. Create a POOL for every source-destination pair and vehicle category.

POOL DEFINITION
- A pool is a group of 5 alternative paths for the same (source, destination, vehicle category).
- The 5 paths must be feasible and sorted from best to worst.
- I need pair-level pools and also a network-level view of pooled alternatives / shared-edge usage.

B. BLOCKAGE / DISRUPTION RESPONSE
When a sudden block appears, a link breaks, or a protest closes a road:
- all source-destination pairs whose chosen paths use that blocked edge become affected,
- rerouting must remain category-wise feasible,
- the framework must compare 3 cases.

THREE CASES

CASE 1: Source-based reroute
- Example only: for source 0 and destination 9, if edge 5-9 is blocked, recompute an alternative route from source 0 to destination 9 while avoiding the blocked edge.
- This example is illustrative only; the implementation must work for all source-destination pairs and all relevant categories.

CASE 2: Current-position reroute with congestion increase
- Example only: for source 0 and destination 9, if edge 5-9 is blocked and the vehicle has already reached node 5 or is near the blocked area, increase weight / congestion around that area and reroute from the current node or current local position.
- Again, implement this for all pairs, not only the example.

CASE 3: Pool-based reroute (PROPOSED METHOD)
- Use the precomputed pool when the sudden block occurs.
- Instantly check the 5 candidate paths.
- Remove the paths that contain blocked / infeasible edges.
- Choose the best remaining feasible path.
- Only if the whole pool becomes invalid, do a lightweight pool repair or pool refresh.
- This case should be designed to reduce online reroute iterations, computation time, and computational power relative to case 1 and case 2.
- But do not claim superiority without measured experiments.
